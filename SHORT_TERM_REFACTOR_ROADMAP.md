# 短期重构路线图 + Issue 清单

## 范围

本路线图聚焦未来 4 到 8 周内最值得投入的重构工作，不追求“大而全”，优先解决二次开发效率、协议适配成本和调试闭环问题。

## 当前基线

本轮 P0 已经完成的内容：

- 真实运行时执行链路已接入主库。
- 调度器已从桩执行切换为真实执行器。
- TCP/UDP/Serial 基础通道已可编译并可接入执行器。
- 协议配置已引入统一定义入口 `DefinitionJson`。
- Demo 和 DeviceMock 均已恢复到可编译状态。

本轮 P0 尚未完成的内容：

- 自动重试、退避、完整状态机仍未落地。
- 自动化回归测试尚未建立。
- 文档、模板、README 编码与一致性治理尚未完成。
- 真实设备联调结果尚未固化为测试资产。

## 总体策略

### 第一原则

- 先把现有协议做实，再扩协议数量。
- 先让模板和运行时一致，再做可视化配置。
- 先建立自动化回归，再做大规模接入。

### 预期结果

完成本轮重构后，项目至少应具备以下能力：

- 新增一个同类硬件协议时，不需要跨 4 到 5 层改代码。
- 协议模板可以直接驱动解析，而不是靠隐藏字符串键拼接。
- Demo/Mock 可以作为真实联调工具和回归测试平台使用。
- 主要协议至少有稳定的自动化回归覆盖。

## 短期路线图

### 阶段 1：主链路收尾

目标：

- 把已经打通的运行时链路补齐为“可稳定联调”的 P0 完成态。

交付物：

- 统一的超时、取消、断线、重连状态机
- 请求响应关联增强
- UDP 会话识别收尾
- 串口轮询、帧间隔与多从站能力收尾

完成标准：

- 至少一个 Modbus TCP 设备可稳定完成真实连接、请求、响应、解析、写入。
- 至少一个 Modbus RTU 设备可稳定完成轮询采集。
- S7 / IEC104 至少完成一次真实联调验证。

### 阶段 2：协议模型统一

目标：

- 统一模板、配置中心、运行时解析器的协议配置模型。

交付物：

- 统一协议 Schema
- 模板加载与运行时解析共享同一模型
- `ProtocolParserFactory` 更完整的注册与选择机制

完成标准：

- 模板文件可以直接加载并用于运行时解析，不再依赖手工塞 `ParseRules`
- 新增协议不需要修改多个不一致的配置结构

### 阶段 3：测试闭环建立

目标：

- 让 `DeviceMock`、Demo、自动化测试形成闭环。

交付物：

- 协议样例报文库
- 自动化集成测试
- 回放测试能力
- 异常帧回归用例

完成标准：

- 每次修改 parser 后，可通过一键测试验证至少 Modbus、S7、自定义协议的关键路径
- 能复现 CRC 错误、长度错误、异常码等典型问题

### 阶段 4：开发体验补齐

目标：

- 把项目变成“可持续扩展”的开发平台，而不是 Demo 集。

交付物：

- 真实协议调试工具
- 模板版本管理
- 配置合法性校验
- 统一地址/点位模型

完成标准：

- 二开人员可以通过模板、调试工具和回归测试完成协议接入，而不是大量读源码试错

## 风险控制

### 风险 1：边重构边扩协议

风险：

- 新增协议会把现有不一致的模型进一步固化，后续更难收敛。

处理：

- 本轮只允许补现有协议能力，不优先加新协议家族。

### 风险 2：继续保留桩实现对外展示

风险：

- 使用者误以为功能已完成，导致错误预期和后续维护成本上升。

处理：

- 对所有桩实现加明确标识，或在本轮内替换为真实实现。

### 风险 3：缺少自动化回归

风险：

- 任一 parser 变更都可能造成隐性回归，且难以快速定位。

处理：

- `DeviceMock` 和样例报文回放必须成为本轮正式交付的一部分。

### 风险 4：构建环境不稳定

风险：

- 当前 .NET 10 / MSBuild 环境存在 workload resolver 假失败，且并行构建容易触发文件锁。

处理：

- 短期内统一采用串行构建。
- 在脚本和文档中固化 `MSBuildEnableWorkloadResolver=false`。
- 后续单独治理构建环境问题，避免与协议重构任务混杂。

## Issue 清单

以下条目可直接拆分为仓库 Issue。

### P0-001 主链路收尾：超时、取消、重连状态机

优先级：P0

目标：

- 在现有执行链路上补齐统一状态机，避免链路虽然已打通但行为不稳定。

范围：

- `src/Vktun.IoT.Connector`
- `src/Vktun.IoT.Connector.Business`
- `src/Vktun.IoT.Connector.Communication`
- `src/Vktun.IoT.Connector.Serial`

验收标准：

- 请求超时、主动取消、设备断线、重连四类场景有一致行为。
- 错误可回传到 `CommandResult` / 事件层。

### P0-002 调度器增强：失败重试与优先级执行

优先级：P0

目标：

- 在真实执行器基础上补齐失败重试、退避和优先级能力。

范围：

- `src/Vktun.IoT.Connector.Concurrency`

验收标准：

- 任务支持可配置重试次数和退避策略。
- 优先级不仅保存在字段中，而会影响实际执行顺序。

### P0-003 协议配置统一收尾

优先级：P0

目标：

- 从“引入 `DefinitionJson`”推进到“模板与运行时完全对齐”。

范围：

- `src/Vktun.IoT.Connector.Core`
- `src/Vktun.IoT.Connector.Configuration`
- `src/Vktun.IoT.Connector.Protocol`

验收标准：

- 解析器可直接消费统一模型。
- 新模板不再依赖旧式隐式 `ParseRules` 键。

### P0-004 UDP / Serial 运行时能力收尾

优先级：P0

目标：

- 让当前已接入的 UDP / Serial 通道从“基础可用”提升到“稳定可联调”。

范围：

- `src/Vktun.IoT.Connector.Communication`
- `src/Vktun.IoT.Connector.Serial`

验收标准：

- UDP 可稳定识别设备与会话。
- Serial 可支持轮询、帧间隔、超时和多从站场景。

### P0-005 真实设备联调固化

优先级：P0

目标：

- 把真实联调结果沉淀为文档、样例和测试输入，而不是只停留在人工验证。

范围：

- `demo/Vktun.IoT.Connector.Demo`
- `demo/Vktun.IoT.Connector.DeviceMock`
- 文档与模板目录

验收标准：

- 至少沉淀一套 Modbus TCP 联调样例。
- 至少沉淀一套 Modbus RTU 联调样例。

### P1-001 建立协议自动化回归测试

优先级：P1

目标：

- 建立 parser 与协议样例的自动回归体系。

范围：

- 新增测试工程
- `demo/Vktun.IoT.Connector.DeviceMock`

验收标准：

- 覆盖正常帧、短帧、异常码、CRC 错误、长度错误
- 修改 parser 后可一键回归

### P1-002 升级 DeviceMock 为协议回归平台

优先级：P1

目标：

- 从“演示模拟器”升级为“联调与测试基础设施”。

范围：

- `demo/Vktun.IoT.Connector.DeviceMock`

验收标准：

- 可配置启动多协议模拟设备
- 支持固定报文、动态数据、错误注入
- 可用于集成测试

### P1-003 设计统一地址 / 点位 DSL

优先级：P1

目标：

- 降低不同协议下点位建模和解析差异。

范围：

验收标准：

- 对 Modbus / S7 / 自定义协议至少形成一套统一地址表达
- 支持地址标准化、校验和序列化

### P1-004 做出真实可用的协议调试工具

优先级：P1

目标：

- 把当前客户端从“假读假写”升级为真实联调工具。

范围：

- `demo/Vktun.IoT.Connector.Client`

验收标准：

优先级：P2

目标：

- 解决当前 workload resolver 假失败和并行构建锁文件问题。

范围：

- 构建脚本

- CI 配置

验收标准：

- 不依赖人工设置也能稳定构建。
- CI 与本地构建参数一致。

### P2-002 文档与模板一致性治理

优先级：P2

目标：

- 解决文档、模板、代码认知不一致问题。

范围：

- 根目录 README
- `src/Vktun.IoT.Connector.Protocol/Templates`
- `demo` 文档

验收标准：

- 编码统一
- 示例可运行
- 桩实现标识清晰

## 不建议本轮优先做的事

- 继续横向扩更多协议种类
- 过早开发大型 Web 管理后台
- 过早宣传高并发与高可用指标
- 在没有真实回归体系前做大范围 parser 重写

## 建议的 Issue 使用方式

- P0 按单条 Issue 独立跟踪，必须绑定验收标准。
- P1 可按“能力包”拆 Issue，再在里程碑下聚合。
- P2 可与文档治理、测试治理并行推进。
