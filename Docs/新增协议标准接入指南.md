# 新增协议标准接入指南

本文档定义新增工业协议时的标准工作流。目标是让协议能力同时具备代码实现、配置模板、测试回归和用户文档，而不是只增加一个解析器类。

## 接入准入

新增协议进入主 README 前，应满足以下条件：

- 有明确的 `ProtocolType` 或插件注册标识。
- 有解析器实现，并注册到 `ProtocolParserFactory`。
- 有标准模板，模板能映射到 `ProtocolConfig.DefinitionJson` 或兼容 `ParseRules`。
- 有最小请求/响应样例和单元测试。
- 有配置字段说明和故障排查说明。

## 标准步骤

1. 定义协议身份

   - 内置协议：在 `ProtocolType` 增加枚举值。
   - 插件协议：使用唯一 `ProtocolId`，避免污染核心枚举。
   - 明确 `ProtocolVersion`、`Vendor`、`DeviceModel` 和 `TemplateSource`。

2. 实现解析器

   - 实现 `IProtocolParser`。
   - 解析结果必须输出 `DeviceData`，并填充 `DeviceId`、`ChannelId`、`ProtocolType`、`CollectTime`、`DataItems`。
   - 可恢复错误写入 `ErrorMessage` 或测试断言，不吞异常。
   - 指令下发优先提供强类型命令打包方法；如果通用入口不可用，异常消息必须说明替代 API。

3. 注册解析器

   - 在 `ProtocolParserFactory` 注册解析器和元数据。
   - 元数据至少包含名称、版本、适用设备、支持能力、作者和状态。
   - 对同一协议的多版本或多设备型号，优先使用 `ProtocolType + ProtocolVersion + DeviceModel` 匹配。

4. 建立模板

   - 模板放在 `src/Vktun.IoT.Connector.Protocol/Templates/` 或 `templates/`。
   - 模板字段必须符合 `协议模板字段说明.md`。
   - 新模板需要能通过配置校验报告，不允许只放示意 JSON。

5. 增加测试

   - 正常响应解析测试。
   - 空帧、短帧、长度错误、校验错误或协议异常码测试。
   - 模板加载和兼容性测试。
   - 如涉及通道交互，增加 DeviceMock 或本机集成测试。

6. 更新文档

   - 在协议 README 中列出支持能力和限制。
   - 在模板说明中补充字段。
   - 如果是 NuGet 对外能力，补充最小可运行示例。

## 推荐目录

```text
src/Vktun.IoT.Connector.Protocol/Parsers/{ProtocolName}Parser.cs
src/Vktun.IoT.Connector.Protocol/Templates/{ProtocolName}Template.json
tests/Vktun.IoT.Connector.ProtocolTests/{ProtocolName}RegressionTests.cs
Docs/{ProtocolName}协议接入说明.md
```

## 完成定义

一个协议接入完成后，应能回答以下问题：

- 使用哪个通道连接设备？
- 使用哪个模板描述点位？
- 如何构造一次采集命令？
- 采集结果中每个点位的类型、单位、质量位从哪里来？
- 解析失败时用户能看到什么错误？
- 是否有可重复执行的自动化回归用例？
